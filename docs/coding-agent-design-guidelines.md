# CodingAgent Design Guidelines

このドキュメントは、`/Users/s01080/src/gh-me/zonu-dev/app-pages` で実装している ZOOCHI 系静的ページのデザインと言語運用を、他プロダクトでも CodingAgent が再現できるようにするための実装ガイドです。

## 目的

- ZOOCHI 系ページの見た目を、別アプリや別文書ページでも一貫して再現する
- 単なる色合わせではなく、余白、角丸、影、動き、コピーのトーンまで揃える
- 新規ページ追加時に、既存のトップページ、サポート、プライバシーポリシーと同じブランド体験を保つ

## デザインの核

このリポジトリのデザインは、以下の性格を必ず維持すること。

- 明るいライトテーマを前提にする
- 業務システム風ではなく、親しみがあり少しおもちゃ的な UI にする
- 文字は太め、面は丸く、境界線ははっきり、影は短く厚めにする
- ベースはニュートラルカラーで整え、製品ごとの差はアクセントカラーで出す
- 動きは控えめだが、完全に無機質にはしない
- コンテンツはユーザー向けに簡潔に書き、開発事情やリポジトリ事情は表に出さない

## 非交渉ルール

- ダークモード前提にしない
- ガラスモーフィズム、薄いグレー 1px 線、無彩色フラット UI に寄せない
- アクセント色をページ内で乱立させない
- 角丸を小さくしすぎない。カード系は `1rem` 未満にしない
- CTA をベタ塗り常時強調にしない。通常時は白基調、ホバー時にテーマ色へ反転する
- アニメーションは短く軽くする。長いフェードや大きい移動は使わない
- JavaScript 依存の派手な演出を足さない。基本は静的 HTML + CSS で完結させる

## 基本トークン

必ず CSS カスタムプロパティで管理し、共有スタイルに寄せること。新規プロダクトでもまずこの形から始める。

```css
:root {
    --font-family-base: 'Zen Maru Gothic', 'Hiragino Sans', 'Yu Gothic UI', sans-serif;

    --bg: #f4f7f9;
    --bg-dot: rgba(148, 163, 184, 0.34);
    --surface: #ffffff;
    --surface-soft: #f8fafc;
    --surface-dark: #1e293b;

    --text: #1e293b;
    --muted: #64748b;
    --line: #dbe4ec;
    --line-strong: #cbd5e1;

    --shadow: rgba(148, 163, 184, 0.24);
    --shadow-dark: rgba(15, 23, 42, 0.18);
    --button-shadow: #e2e8f0;
}
```

### 背景

- `body` 背景は `--bg` に加えて、ドット状の `radial-gradient` を重ねる
- さらに `body::before` で淡い白グラデーションを重ね、完全なフラット面にしない
- 背景はあくまで薄く、主役はカード面と本文にする

### 文字

- ベースフォントは `Zen Maru Gothic`
- 見出しは `font-weight: 900`
- 本文は `700` を基準にし、軽いウェイトは基本的に使わない
- 見出しは詰め気味の letter-spacing を使って、丸い書体でも締まりを出す

### 角丸と線

- 言語切替やメタ情報のピル: `999px`
- ボタン、ステータス、リストリンク: `1rem` 前後
- カード内サブ面: `1.5rem`
- カード外枠、文書シェル、フッター面: `2rem` から `2.5rem`
- 線は `2px` か `3px`。極細にしない

### 影

- 基本は `0 4px 0` または `0 6px 0` の段差影
- ぼかし中心ではなく、軽い立体感を出すための短いオフセット影にする
- 押下時は `translateY(4px)` で影を消し、物理ボタン風に見せる

## テーマ設計

ページごとの差分は、テーマ用変数だけで作る。コンポーネントごとに色を直接焼き込まない。

```css
.theme-home {
    --theme-accent: #38bdf8;
    --theme-accent-deep: #0f172a;
    --theme-soft: #e0f2fe;
    --theme-soft-border: #bae6fd;
    --theme-ink: #0369a1;
}

.theme-rooted {
    --theme-accent: #34d399;
    --theme-accent-deep: #047857;
    --theme-soft: #ecfdf5;
    --theme-soft-border: #bbf7d0;
    --theme-ink: #047857;
}

.theme-spacepin {
    --theme-accent: #6366f1;
    --theme-accent-deep: #4f46e5;
    --theme-soft: #eef2ff;
    --theme-soft-border: #c7d2fe;
    --theme-ink: #4f46e5;
}
```

### テーマの使い方

- 通常面や本文はニュートラルカラーを維持する
- 製品差分は次の要素に限定して出す
- ブランドピル
- セクション番号バッジ
- FAQ やタグの淡色面
- CTA のホバー色
- 必要ならアプリアイコン風バッジのグラデーション

## レイアウト原則

### 共通

- 1 カラム中心で、横幅は狭めに保つ
- 行間は広めに取り、詰め込みすぎない
- 余白は `16`, `20`, `24`, `32`, `40` を基準にする
- `320px` 幅でも崩れないことを最低条件にする

### トップページ

- `body.home-page` は中央寄せの縦レイアウト
- コンテナ幅は `56rem` 程度を上限にする
- 上から `page-header`、`app-overview`、`site-footer`
- 製品一覧はモバイル 1 カラム、`768px` 以上で 2 カラム

### 文書ページ

- `.document-page .container` は `768px` 程度を上限にする
- 上から `document-toolbar`、`document-header`、`document-shell`、`doc-footer`
- 文書本体は大きな白カード 1 枚にまとめ、その中を番号付きセクションで分割する

## コンポーネント再現ルール

### 1. 言語切替

- `select` ベースで実装する
- 見た目はピル型、右端に小さなシェブロンアイコンを重ねる
- select の左に丸型の国旗アイコンを表示する。アイコンは [circle-flags](https://github.com/HatScripts/circle-flags)（MIT）の SVG を `assets/flags/` に配置して使う
- option のテキストは言語コード（EN, JA, KO, ZH, VI, ID）のみ。言語名は表示しない
- 言語切替時に JS で国旗画像の `src` を切り替える（`data-flag` 属性を参照）
- 表示幅は国旗と選択中ラベルに合わせて JS で自動調整する
- すべての公開ページで、同じ言語セットを同じ順序で並べる

対象ロケール:

| コード | 国旗ファイル |
|--------|-------------|
| `en` | `us.svg` |
| `ja` | `jp.svg` |
| `ko` | `kr.svg` |
| `zh-Hans` | `cn.svg` |
| `vi` | `vn.svg` |
| `id` | `id.svg` |

### 2. ヒーロー見出し

- 大見出しは 1 行で無理に説明を詰めない
- ブランド名や製品名は、濃色の角丸チップで見せてもよい
- サイズは `2.25rem` から始め、広い画面では `3rem` まで拡大する

### 3. アプリカード

- 外枠: 白面、`3px` ボーダー、太い角丸、短い段差影
- 上部ヒーロー面: `surface-soft` を使った内側カード
- 左にバッジ、右に製品名と短い説明
- 下部に `status-panel` と `action-button` を縦積みする
- バッジは文字 1 文字（R, A, P）またはアプリアイコン画像（`<img class="app-badge">` + `padding` で調整）
- アプリアイコン画像を使う場合、背景グラデーションはそのまま維持し `object-fit: contain` で収める
- カードの並び順はモバイル 1 カラム時の表示順で決める（現在: Auto → Palette → SpacePin → Rooted）

構造例:

```html
<!-- 文字バッジ -->
<div class="wobble-target app-badge app-badge--rooted">R</div>

<!-- 画像バッジ -->
<img class="wobble-target app-badge app-badge--spacepin" src="assets/spacepin-icon.png" alt="Space Pin">
```

### 4. ステータスパネル

- 説明用の小面として使う
- 背景は `surface-soft`
- 左にラベルとバッジ、右に丸アイコン
- 状態バッジは小さく、本文より一段弱い色にする

### 5. アクションボタン

- 通常時は白地
- ホバー時に製品色へ面ごと切り替え、ラベルを白へ反転する
- 右アイコンは統一して右矢印（`→`）の SVG を使う。個別のアイコン（再生、ダウンロード等）は使わない
- クリック感のある `toy-btn` を付与する
- 影は `0 4px 0 #cbd5e1` で統一。ホバー `translateY(-1px)` + `0 5px 0`、押下 `translateY(4px)` + 影消し

### 6. フッターの問い合わせ面

- フッターは濃色面 1 枚で締める
- 背景は `surface-dark`
- 右上に薄い円形オーブを置いて単調さを避ける
- 問い合わせリンクは白カード 2 枚程度を並べる
- それぞれに色つきアイコンボックスを持たせる
- 権利表記（copyright）は問い合わせカードの外、`site-footer` 直下に右揃えで置く。罫線は付けない

### 7. 文書ヘッダー

- CSS Grid で構成。アイコンが左列（2 行分）、タイトルと更新日が右列に縦並び
- ブランドバッジ（SPACEPIN 等）は使わない。アプリアイコンバッジで製品を識別する
- 更新日はラベル（「最終更新日:」等）を省略し、日付のみ小さく表示する（`font-size: 0.78rem`、ボーダーなし、背景なし）
- タイトルは `1.75rem`、アイコンの視覚中心と揃える

### 8. 文書セクション

- 各セクションは左に番号、右に見出し
- 本文は番号行から少し下げて開始する
- 箇条書きは淡い面にまとめ、通常のブラウザ既定の箇条書きを使わない
- 先頭マーカーは小さな丸とソフトシャドウで表現する
- セクション間の区切り線は `2px dashed rgba(148, 163, 184, 0.36)` の破線を使う
- 機能一覧は `feature-tags` ではなく通常の箇条書き（`<ul>`）で統一する

構造例:

```html
<section class="doc-section">
  <div class="doc-section__heading">
    <span class="doc-section__number">3</span>
    <h2>Main Features</h2>
  </div>
  <div class="doc-section__body">
    <p>説明文</p>
    <ul>
      <li>項目 A</li>
      <li>項目 B</li>
    </ul>
  </div>
</section>
```

### 9. FAQ

- FAQ は `.qa-list` 内に `.qa-card` を並べる
- Q と A はそれぞれ丸バッジで識別する
- カード背景は `surface-soft`
- ボーダーは本文シェルより弱くし、内部の情報整理用カードに見せる

### 10. CTA セクション

- 文書下部で関連文書へ誘導するときに使う
- 単独リンクを本文中に裸で置かず、`doc-section doc-section--cta` として 1 セクション扱いにする
- リンクはトップページと同じ `action-button` スタイル（白地、右矢印、ホバーでテーマ色反転）を使う

### 11. ツールバー（文書ページ上部）

- トップページへ戻るロゴリンク、言語セレクト、表示切替ボタンを 1 行に横並びする
- 左端は ZOOCHI ロゴ画像を使ったトップページリンクにする
- 3 要素の影・ホバー・押下アニメーションは `toy-btn` と同じ値で統一する（`0 4px 0 #cbd5e1`）
- ロゴリンクは左寄せ、言語セレクトと表示切替は右寄せ

### 12. PC/Mobile 表示切替

- デスクトップブラウザに表示切替ボタンを設ける（タッチデバイスでは非表示: `@media (hover: none) and (pointer: coarse)`）
- Mobile モード時はコンテナ幅を `375px` に制限し、1 カラムレイアウトにする
- `?view=mobile` パラメータで状態を保持し、ページ遷移・言語切替時に引き継ぐ
- Mobile モード時は全内部リンクに `?view=mobile` を自動付与する

## 動きのルール

使ってよい動きは次の程度まで。

- ページロード時の `rise-in`
- バッジやアイコンの軽い `bob`
- ホバー時の `icon-wobble`
- ボタン押下時の `translateY(4px)`
- クリック時の `scale(0.85) + rotate`（wobble-container 内の要素）

### アニメーションの原則

- 0.6 秒以下
- 移動量は小さく
- 回転は数度から十数度以内
- ループする動きは 1 箇所か 2 箇所まで
- `prefers-reduced-motion: reduce` を必ず尊重する

### wobble のスコープ

- `wobble-container:hover` は直接ホバーした要素だけを揺らす
- `app-card:hover` はカード全体のホバーでバッジを揺らす
- フッター（`footer-card`）では、セクション全体のホバーで見出しアイコン（`.contact-panel__heading-icon`）だけを揺らし、子ボタンのアイコンは各ボタン（`.btn-wobble-group`）のホバーで個別に揺らす
- クリック（`:active`）時の回転角度は CSS 変数 `--wobble-rotate` で制御し、JS の `mousedown` イベントで毎回 -15deg〜+15deg のランダム値を設定する

## コピーと情報設計

- 見出しは短く、説明責務を明確にする
- 製品説明は 1 から 2 行で要点を言い切る
- サポート文書は「何ができるか」「どう使うか」「困ったらどうするか」の順で整理する
- ポリシー文書は法務文体に寄せすぎず、平易な説明を保つ
- 公開ページ上で、リポジトリ名、実装都合、内部フロー、開発メモを出さない
- セクション見出しは日本語に統一する（「PRODUCTS」ではなく「つくったもの」）。英語と日本語を混在させない
- ボタンラベルもひらがなの柔らかいトーンに揃える（「使う」ではなく「つかってみる」）
- 更新日のラベル（「最終更新日:」等）は省略し、日付のみ表示する

## 実装スタイル

- 共有ルールは `assets/style.css` に追加する
- ページ単位の見た目差分を増やしすぎない
- まず意味的な HTML を組み、その後クラスで見た目を付ける
- SVG アイコンはシンプルな線画または単色面で揃える
- 矢印やナビゲーション系アイコンは CSS `mask-image` + インライン SVG data URI で実装し、テキスト文字（`←` `→` `›`）は使わない
- フォーカスリングは削除しない
- `target="_blank"` を使う外部リンクには `rel="noopener noreferrer"` を付ける

## このリポジトリ内で新規公開ページを作る場合の追加ルール

- トップページは日本語が `index.html`、英語が `index.en.html`
- 文書ページは英語が `index.html`、他言語が `index.<locale>.html`
- すべての公開ページで `en`, `ja`, `ko`, `zh-Hans`, `vi`, `id` を同時に更新する
- 言語切替、トップへの戻りリンク、フッター戻り導線は全ロケールで揃える
- 公開パスを変える場合でも、`apps/*.html` の旧リダイレクトページは残す

## やってはいけないこと

- 製品ごとに完全に別 UI を作ること
- ヒーローに大きな写真や複雑なイラストを入れて主張を強くしすぎること
- 本文カードの外側にさらに複数の強い装飾を重ねること
- CTA を複数色のグラデーションボタンだらけにすること
- 小さな画面で横スクロールを発生させること
- 公開ページに開発者向け用語を出すこと

## 実装チェックリスト

- ライトテーマ、丸い面、太い文字、短い段差影になっている
- 背景は薄いドットパターン付きのニュートラル面になっている
- 製品差分はテーマ変数中心で出している
- CTA は通常時白基調、ホバー時テーマ色反転になっている
- モバイルでは 1 カラム、PC では必要に応じて 2 カラムになっている
- `prefers-reduced-motion` を壊していない
- ロケールセットとリンク導線が揃っている
- 文言がユーザー向けで、内部事情を含んでいない

## 迷ったときの優先順位

1. 既存の `assets/style.css` と同じパターンを再利用する
2. 新しい見た目を足すより、既存コンポーネントの組み合わせで解決する
3. 製品差分は色と文言で出し、骨格は変えない
4. 派手さより、一貫性と読みやすさを優先する
